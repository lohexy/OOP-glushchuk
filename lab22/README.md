Лабораторна робота №22: Принцип підстановки Лісков (LSP)

Тема: Виявлення порушень LSP та проектування альтернативних рішень

Мета:
Навчитися ідентифікувати порушення принципу LSP в ієрархіях класів та застосовувати рефакторинг (зміна ієрархії, інтерфейси) для створення стабільного та гнучкого коду.

1. Початкова ієрархія (Порушення LSP)

У початковому варіанті ми маємо базовий клас `Printer`, який передбачає можливість кольорового друку. Клас `BlackAndWhitePrinter` наслідує його, але оскільки він не підтримує кольори, він порушує очікувану поведінку.

Приклад коду з помилкою:
public class Printer {
    public virtual void PrintColorImage(string content) {
        Console.WriteLine("Друк кольорового зображення: " + content);
    }
}

public class BlackAndWhitePrinter : Printer {
    public override void PrintColorImage(string content) {
        // Порушення LSP: підклас не може виконати дію базового класу
        throw new NotSupportedException("Чорно-білий принтер не підтримує кольоровий друк!");
    }
}

Аналіз порушення:

1. Порушення контракту: Базовий клас `Printer` гарантує клієнту виконання кольорового друку. Підклас `BlackAndWhitePrinter` не виконує цю обіцянку.
2. Runtime Error: Клієнтський код, який працює з типом `Printer`, впаде під час виконання, якщо йому передадуть чорно-білу модель.
3. Прихована логіка: Розробнику доводиться використовувати перевірки на кшталт `if (p is not BlackAndWhitePrinter)`, що суперечить принципам ООП.

2. Рефакторинг (Дотримання LSP)

Для виправлення ми використали розділення інтерфейсів (Interface Segregation). Тепер ми не змушуємо класи реалізовувати те, чого вони не вміють.

 Виправлений код:

// Загальний інтерфейс для базових функцій
public interface IPrinter {
    void PrintGreyScale(string content);
}

// Спеціалізований інтерфейс для кольорового друку
public interface IColorPrinter : IPrinter {
    void PrintColor(string content);
}

// Реалізація тільки базових функцій
public class BasicPrinter : IPrinter {
    public void PrintGreyScale(string content) => 
        Console.WriteLine($"[B&W] Друк: {content}");
}

// Реалізація всіх функцій (кольоровий принтер)
public class HighEndPrinter : IColorPrinter {
    public void PrintGreyScale(string content) => 
        Console.WriteLine($"[B&W Mode] Друк: {content}");

    public void PrintColor(string content) => 
        Console.WriteLine($"[Color Mode] Друк: {content}");
}

Чому це рішення вірне:

Передбачуваність: Метод, який потребує кольорового друку, тепер приймає `IColorPrinter`. Це гарантує, що туди не потрапить чорно-білий принтер.
Безпека: Помилки відловлюються на етапі написання коду (компіляції), а не під час роботи програми.
Чистота: Кожен клас відповідає лише за те, що він реально може зробити.

3. Висновки

Дотримання LSP дозволяє створювати ієрархії, де підкласи справді є розширенням базового типу, а не його обмеженням. Використання інтерфейсів замість глибокого наслідування допомагає уникнути "небезпечних" методів, які викидають виключення, і робить код надійнішим.
